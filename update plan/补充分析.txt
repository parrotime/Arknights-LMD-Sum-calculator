================================================================================
            龙门币凑数计算器 2.0 改造计划 —— 角度四：补充分析
================================================================================

本文件补充前三份分析（代码层面、用户体验、算法优化）未涉及的维度。


【一、安全性】

1. 没有使用 helmet 中间件（重要）
   现状：Express 服务器没有设置任何安全响应头。
   缺少 X-Content-Type-Options、X-Frame-Options、
   Content-Security-Policy、Strict-Transport-Security 等头部。

   建议：安装并使用 helmet 中间件，一行代码即可添加常见安全头。

2. CORS 允许了不必要的 HTTP 方法
   位置：server.js 第 50 行
   现状：methods 配置为 ["GET", "POST", "PUT", "DELETE"]，
   但实际只用到了 GET（测试路由）和 POST（/find-paths）。
   允许 PUT 和 DELETE 增加了攻击面。

   建议：改为 methods: ["GET", "POST"]。

3. rate limiter 重复注册（bug，已在代码分析中提及，此处补充安全视角）
   位置：server.js 第 31 行和第 56 行
   问题：同一个 apiLimiter 对 /find-paths 注册了两次。
   这不会导致限制加倍（Express 中间件按顺序执行，
   两次都会消耗计数），但行为不可预测，可能导致
   实际限制与预期不符。

4. items 数组元素缺少深度校验
   位置：server.js 第 77~86 行
   现状：只检查了 items 是否为数组、是否为空、长度是否 ≤300。
   但没有校验每个 item 对象的结构（是否有 id、item_value 等字段，
   字段类型是否正确）。恶意用户可以发送畸形数据。

   建议：对 items 数组中的每个元素进行 schema 校验，
   或者更好的方案是：前端不发送 items，后端自行从 DataService 获取。

5. 缓存 key 包含完整 items JSON（潜在 DoS）
   位置：server.js 第 147 行
   现状：cacheKey = `paths:${target}:${JSON.stringify(items)}:...`
   用户可以发送略有不同的 items 数组来绕过缓存，
   每次都触发完整计算，消耗服务器资源。
   同时巨大的 cacheKey 字符串本身也浪费内存。

   建议：如果改为后端自行过滤 items，cacheKey 只需包含
   target + 开关状态的哈希，极大简化且更安全。

6. 请求体大小限制过大
   位置：server.js 第 54 行 `express.json({ limit: "1mb" })`
   现状：1MB 对于这个 API 来说过大（正常请求约 30KB）。

   建议：降低到 50KB 或 100KB。


【二、测试】

1. 零测试覆盖（严重）
   现状：整个项目没有任何测试文件。package.json 中配置了
   "test": "craco test"，但没有任何测试用例。
   对于一个涉及复杂数学计算的工具，没有测试意味着：
   - 无法验证算法正确性
   - 重构时无法确保不引入回归 bug
   - 无法验证边界情况（target=0、target=±5000、空 items 等）

   建议按优先级添加：
   a) 算法单元测试（最重要）：
      - findPaths 对已知输入的输出是否正确
      - 边界值测试：target=0、±1、±5000
      - 贪心组合器的正确性验证
      - savePath 的去重和限制逻辑
   b) API 集成测试：
      - /find-paths 的正常请求和异常请求
      - 输入校验是否生效
      - 缓存命中/未命中
      - 超时处理
   c) 前端组件测试：
      - PathRenderer 渲染是否正确
      - 输入校验逻辑
      - 设置开关的状态管理

2. 没有 CI/CD 中的自动化测试
   建议：在 GitHub Actions 中配置 push/PR 时自动运行测试。


【三、部署与运维】

1. 环境配置全部硬编码（严重）
   现状：
   - CORS origin 通过注释切换（server.js 第 43~49 行）
   - 前端 API 地址通过注释切换（Transmission.js 中）
   - 端口号硬编码为 3002（server.js 第 29 行）
   每次部署/本地调试都需要手动改注释，极易出错。

   建议：使用环境变量 + .env 文件：
   - .env.development: REACT_APP_API_URL=http://localhost:3002
   - .env.production: REACT_APP_API_URL=https://api.ark-lmd.top
   - 后端同理：CORS_ORIGIN、PORT 等从 process.env 读取

2. 没有进程管理
   现状：后端直接用 node 启动，如果崩溃不会自动重启。

   建议：使用 PM2 或 systemd 管理后端进程，
   配置自动重启、日志轮转、内存限制。

3. 没有错误监控
   现状：错误只打印到 console.log/console.error，
   服务器崩溃或异常无法及时发现。

   建议：接入免费的错误监控服务（如 Sentry 免费版），
   前后端都可以使用，异常时自动通知。

4. 日志不规范
   现状：使用 console.log 打印日志，没有日志级别区分，
   没有结构化格式，大量被注释掉的 console.log 散落各处。

   建议：使用轻量日志库（如 pino），区分 info/warn/error 级别，
   输出 JSON 格式便于后续分析。

5. deploy 脚本使用 gh-pages 但实际部署在自定义域名
   位置：package.json 第 20 行
   现状：deploy 命令是 "gh-pages -d build"，
   但 homepage 是 "https://ark-lmd.top"。
   如果实际部署方式已经不是 gh-pages，这个脚本就是误导性的。

   建议：确认实际部署流程，更新或移除过时的 deploy 脚本。


【四、SEO 与可发现性】

1. HTML lang 属性错误
   位置：public/index.html 第 2 行 `<html lang="en">`
   现状：网页内容全部是中文，但 lang 设为 "en"。
   这会影响搜索引擎对页面语言的判断，也影响屏幕阅读器的发音。

   建议：改为 `<html lang="zh-CN">`。

2. 缺少 Open Graph 和 Twitter Card 标签
   现状：没有 og:title、og:description、og:image 等标签。
   用户在社交媒体（微博、QQ、B站动态）分享链接时，
   无法显示预览卡片，只会显示一个裸链接。

   建议：添加 Open Graph 标签：
   <meta property="og:title" content="龙门币凑数路径计算器" />
   <meta property="og:description" content="明日方舟龙门币凑数计算工具" />
   <meta property="og:image" content="https://ark-lmd.top/LMD.webp" />

3. manifest.json 仍是 CRA 默认值
   位置：public/manifest.json
   现状：short_name 是 "React App"，name 是 "Create React App Sample"。
   如果用户将网页添加到手机主屏幕，显示的名称是 "React App"。

   建议：改为 "龙门币计算器" / "明日方舟龙门币凑数路径计算器"。

4. apple-touch-icon 指向不存在的文件
   位置：index.html 第 12 行 `href="LMD192.png"`
   现状：public 目录下没有 LMD192.png，只有 logo192.png。
   iOS 用户添加到主屏幕时会显示默认图标。

   建议：修正文件名，或将 logo192.png 替换为项目专属图标。

5. 没有 sitemap.xml
   现状：只有一个基本的 robots.txt（允许所有爬虫），
   但没有 sitemap 帮助搜索引擎发现所有页面。

   建议：生成 sitemap.xml，包含 /、/note、/data、/about 四个页面。

6. SPA 的 SEO 天然劣势
   现状：使用 HashRouter（URL 带 #），搜索引擎对 # 后的内容
   索引能力有限。

   建议：2.0 版本考虑改用 BrowserRouter + 服务端配置 fallback，
   或使用 SSG（如 Next.js 的静态导出）生成静态页面。


【五、前端性能】

1. 没有路由级代码分割
   现状：所有页面（App、Note、Data、About）打包在同一个 bundle 中。
   用户访问首页时，也会加载 Note、Data、About 的代码和数据。
   Data.js 中包含大量硬编码的表格数据，体积不小。

   建议：使用 React.lazy + Suspense 对路由进行懒加载：
   const NotePage = React.lazy(() => import('./pages/Note'));

2. 图片资源没有优化流程
   现状：项目中使用了多张表情包图片（迷迭香系列），
   但没有压缩、没有 WebP 转换、没有响应式图片（srcset）。

   建议：
   a) 将 PNG/JPG 转为 WebP 格式（体积减少 30%~50%）
   b) 提供多尺寸版本，移动端加载小图
   c) 对非首屏图片使用 loading="lazy"

3. CSS 全部全局加载
   现状：App.css（770行）、PathRenderer.css、Data.css 等
   全部是全局 CSS，所有页面都会加载所有样式。
   类名冲突风险高（如 .title 这种通用名）。

   建议：改用 CSS Modules 或 styled-components，
   实现样式的按组件隔离和按需加载。

4. 没有 bundle 分析
   现状：不清楚最终打包体积有多大，哪些依赖占比最高。

   建议：使用 webpack-bundle-analyzer 或 source-map-explorer
   分析打包结果，找出体积优化空间。


【六、数据维护与更新】

1. 游戏数据更新流程缺失（重要）
   现状：所有游戏数据（干员升级消耗、关卡理智、材料合成费用等）
   硬编码在 DataService.js 中。当明日方舟更新（新关卡、新活动、
   数值调整）时，需要手动修改代码并重新部署。

   建议：
   a) 将游戏数据抽取为独立的 JSON 文件
   b) 建立数据更新检查清单（每次游戏大版本更新时对照）
   c) 长远考虑：从 PRTS Wiki API 自动拉取最新数据

2. 前后端数据同步问题
   现状：DataService.js 在前端和后端各有一份副本，
   内容相同但需要手动保持同步。

   建议：（已在代码分析中提及）改为单一数据源。
   如果前端不再发送 items，后端的 DataService 就是唯一数据源。

3. 活动数据的时效性
   现状：活动商店（故事集、sidestory、危机合约）的兑换比例
   是硬编码的。不同活动的兑换比例可能不同。

   建议：在设置面板中允许用户自定义活动商店的兑换比例，
   而不是依赖硬编码值。


【七、PWA 与离线能力】

1. PWA 配置不完整
   现状：有 manifest.json 但内容是 CRA 默认值，
   没有配置 Service Worker（CRA 默认不注册）。
   网页无法离线使用，也无法"安装"到手机主屏幕。

   建议：
   a) 完善 manifest.json（名称、图标、主题色）
   b) 注册 Service Worker，缓存静态资源
   c) 对于计算结果，可以缓存最近的查询供离线查看

2. 没有利用浏览器缓存策略
   现状：静态资源没有配置长期缓存头（Cache-Control）。
   CRA build 会自动给 JS/CSS 加 hash，但需要服务器配合
   设置正确的缓存头。

   建议：确认部署服务器对 build 产物设置了合理的缓存策略。


================================================================================
总结：补充优化优先级排序
================================================================================

第一优先级（必须修复）：
  □ 环境配置改用环境变量（消除注释切换的部署风险）
  □ HTML lang 改为 zh-CN
  □ manifest.json 更新为项目实际信息
  □ 修复 apple-touch-icon 指向不存在的文件
  □ CORS 去掉不必要的 PUT/DELETE 方法

第二优先级（显著提升质量）：
  □ 添加算法核心逻辑的单元测试
  □ 添加 helmet 安全中间件
  □ 降低请求体大小限制
  □ items 数组元素添加 schema 校验（或改为后端自行获取）
  □ 游戏数据抽取为独立 JSON 文件

第三优先级（提升专业度）：
  □ 添加 Open Graph 标签（社交分享预览）
  □ 路由级代码分割（React.lazy）
  □ 图片资源优化（WebP + lazy loading）
  □ 日志规范化（替换 console.log）
  □ 接入错误监控（Sentry 等）

第四优先级（长远规划）：
  □ 完善 PWA 配置 + Service Worker
  □ 考虑 HashRouter → BrowserRouter 迁移
  □ 添加 sitemap.xml
  □ CSS 改为 CSS Modules
  □ 建立游戏数据自动更新机制
  □ 配置 CI/CD 自动化测试和部署
