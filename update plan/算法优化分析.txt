【一、当前算法概述】

核心文件：main/backend/DPnew.js（514 行）

算法类型：动态规划 + 贪心组合优化器
输入：target（目标差值，范围 -5000~5000）、items（可用物品列表）、userLimits（用户限制）
输出：最多 10 条到达目标值的路径（每条路径由若干步骤组成）

执行流程：
  1. 预处理：过滤有效物品，按绝对值降序排列
  2. 第一阶段：枚举每个物品的单步路径（只用一种物品）
  3. 第二阶段：在已有路径基础上，逐步扩展（最多 6 步），尝试添加新物品
  4. 路径规范化：将售卖赤金、基建合成、消耗理智替换为最优组合
  5. 路径去重 + 排序 + 截取前 10 条返回

关键常量：
  MAX_STEPS = 6（最大步骤数）
  MAX_PATHS_PER_SUM = 10（每个金额值最多保存 10 条路径）
  TARGET_PATH_COUNT = 10（目标路径数量，达到后提前终止）

【二、性能瓶颈分析】

1. items.find() 在热循环中反复调用（严重）
   位置：getMaxCountForId、normalizePath、savePath、finalizeResult
   问题：items 数组有 200+ 个元素，items.find() 是 O(n) 线性查找。
   在 savePath 中，一次调用就可能执行 3~4 次 items.find()，
   而 savePath 在双重循环中被频繁调用，总查找次数可达数万次。

   建议：在 findPaths 入口处构建 Map<id, item> 索引表，
   所有查找改为 O(1) 的 Map.get()。预计可减少 30%~50% 的计算时间。

2. 模块级缓存（tradeGoldCache、materialCache、stageCache）永不清理（严重）
   位置：文件顶部第 3~5 行
   问题：这三个 Map 在模块加载时创建，服务器运行期间永不清理。
   随着不同用户的请求积累，缓存会无限增长，最终导致内存泄漏。
   更严重的是：stageCache 的结果依赖于 stageItems 和 items 参数，
   但缓存 key 只用了 subTarget 数值。如果不同用户传入不同的 items
   （因为开关设置不同），会返回错误的缓存结果。

   建议：
   a) 将缓存改为函数内局部变量，每次计算重新创建
   b) 或者将 items 的哈希值纳入缓存 key
   c) 对于 tradeGoldCache 和 materialCache，由于不依赖 items 参数，
      可以保留模块级缓存但设置大小上限（LRU）

3. 搜索空间爆炸（中等）
   位置：第二阶段多步路径搜索（第 458~511 行）
   问题：每一步都遍历所有 dp 状态 × 所有物品 × 所有已有路径，
   组合数量呈指数增长。虽然有 3000 距离剪枝和 enoughPaths 提前终止，
   但在某些输入下（如差值接近 0 或物品种类多时），计算仍然很慢。

   建议：
   a) 按物品价值分层搜索，先用大额物品粗调，再用小额物品微调
   b) 引入更智能的剪枝：如果当前 sum 已经超过 target 很多，
      且剩余物品都是同方向的，可以直接跳过
   c) 考虑双向搜索：从 0 向 target 搜索 + 从 target 向 0 搜索，
      在中间汇合

4. Array.from(dp.entries()) 每步都创建快照（中等）
   位置：第 459 行
   问题：每个 step 迭代开始时，都将整个 dp Map 转为数组。
   随着 dp 中状态增多，这个操作的开销越来越大。

   建议：使用分层 dp，每一步只在上一步的结果基础上扩展，
   避免重复遍历已处理的状态。


【三、正确性问题】

1. 贪心组合优化器可能错过最优解（重要）
   位置：getOptimalTradeGoldCombo、getOptimalMaterialCombo、getOptimalStageCombo
   问题：三个组合优化器都使用贪心策略（从大到小依次填充）。
   对于售卖赤金（2000/1500/1000）和基建合成（400/300/200/100），
   贪心恰好是最优的（因为大面额是小面额的整数倍或接近整数倍）。
   但对于消耗理智（stageCombo），物品价值不规则（72/108/120/144/180/...），
   贪心可能错过最优组合。

   示例：假设 subTarget = 360
   贪心：先用 252（21理智三星），剩余 108，再用 108（9理智三星）= 2 步
   最优：直接用 360（30理智三星）= 1 步
   （注：实际上 stageItems 按价值降序排列，所以 360 会先被尝试，
   但如果 360 对应的物品被用户关闭了，就可能出现次优解）

   建议：对 stageCombo 改用小规模 DP 或完全搜索（物品种类不多，
   最多十几种，完全可以精确求解）。

2. stageCache 缓存污染（bug）
   位置：getOptimalStageCombo（第 95~117 行）
   问题：stageCache 是模块级的，但 getOptimalStageCombo 的结果
   依赖于 stageItems 参数（哪些关卡可用）。不同用户可能开启/关闭
   不同的关卡开关，导致 stageItems 不同。
   第一个用户的计算结果会被缓存，第二个用户即使 stageItems 不同，
   也会命中缓存拿到错误结果。

   建议：要么去掉 stageCache，要么将 stageItems 的 id 列表
   编码进缓存 key。

3. enoughPaths 提前终止不彻底（设计缺陷）
   位置：第 505~509 行（三处被注释掉的 break）
   问题：在第二阶段的三层循环中，enoughPaths 被设为 true 后，
   只有最内层循环（遍历 oldPath）会 break，
   中间两层循环（遍历 items 和 count）的 break 被注释掉了。
   这意味着即使已经找到足够路径，算法仍会继续遍历当前 sum 下的
   所有物品和所有 count 值，浪费大量计算。

   分析：注释掉 break 可能是为了找到更多样化的路径，
   但代价是显著增加计算时间。这是一个质量 vs 速度的权衡。

   建议：至少恢复 items 循环的 break（第 507 行），
   因为同一个 sum 下继续尝试不同物品很少能产生更好的路径。
   或者引入一个"宽限计数器"：找到足够路径后再多搜索 N 次就停止。

4. 路径规范化后金额校验过于宽松（潜在 bug）
   位置：savePath 第 186 行
   问题：epsilon * 100 作为容差，如果 epsilon = 1e-6，
   则容差为 1e-4，这对于整数运算来说几乎没有意义。
   但如果未来 epsilon 被调大，可能导致不精确的路径被接受。

   建议：既然所有物品价值都是整数，应该直接用整数运算，
   去掉 epsilon 相关的浮点容差逻辑。


【四、搜索策略优化】

1. 距离剪枝阈值硬编码为 3000（可优化）
   位置：第 462 行 `if (Math.abs(currentSum - target) > 3000) continue;`
   问题：3000 是一个经验值，对于小差值（如 target=10）过于宽松，
   对于大差值（如 target=5000）可能过于严格。

   建议：改为动态阈值，例如 Math.max(Math.abs(target) * 1.5, 1000)，
   根据目标值自适应调整搜索范围。

2. 物品扩展范围的启发式不够精确
   位置：第一阶段第 431 行和第二阶段第 469 行
   问题：
   第一阶段：Math.abs(itemValue * (count+1) - target) <= Math.abs(target) + Math.abs(itemValue)*0.5
   第二阶段：Math.abs(currentSum + itemValue*(count+1) - target) <= Math.abs(target) + Math.abs(itemValue)
   这两个条件控制每个物品最多使用多少次，但启发式系数（0.5 和 1.0）
   是经验值，可能导致搜索空间过大或过小。

   建议：
   a) 对于第一阶段，如果 itemValue 和 target 同号，
      最大 count 应该是 ceil(|target| / |itemValue|) + 1 即可
   b) 对于第二阶段，考虑剩余步数能覆盖的最大调整量来决定扩展范围

3. 缺少"反向搜索"策略
   问题：当前算法只从 0 向 target 方向搜索。
   对于某些差值，从 target 反向减去物品价值可能更快找到路径。

   建议：实现双向 BFS/DP：
   a) 正向：从 sum=0 开始，逐步加物品
   b) 反向：从 sum=target 开始，逐步减物品
   c) 当两个方向的搜索空间有交集时，合并路径
   这可以将搜索空间从 O(V^n) 降低到 O(2 * V^(n/2))。

4. 没有利用物品价值的数学特性
   问题：很多物品价值是 100 的整数倍（1000/1500/2000/100/200/300/400），
   升级消耗的尾数相对固定。可以利用这个特性进行分层搜索。

   建议：
   a) 第一层：用"尾数物品"（升级消耗，值不是 100 的倍数）确定个位和十位
   b) 第二层：用"整数物品"（赤金/合成/关卡，值是 100 的倍数）补齐百位以上
   这样可以大幅减少组合搜索空间。


【五、结果质量优化】

1. 路径多样性不足
   问题：当前排序标准是"步骤种类数少 > 总物品数少 > ID 排序"。
   这导致返回的 10 条路径可能非常相似（只是某个物品数量差 1）。
   用户更希望看到不同策略的路径（如"纯升级方案""纯刷图方案""混合方案"）。

   建议：
   a) 引入路径"类型标签"：根据路径中物品的 type 分布，
      标记为"升级型""刷图型""合成型""混合型"等
   b) 在选择路径时，优先保证每种类型至少有 1~2 条
   c) 同类型内再按步骤数/总数排序

2. 缺少路径"代价评估"
   问题：当前只比较步骤种类数和总物品数，没有考虑实际游戏代价。
   例如"消耗 36 理智刷一关"和"对 1 名干员使用 1 次作战记录"
   在游戏中的代价完全不同，但算法认为它们都是"1 步"。

   建议：引入加权代价函数：
   a) 理智消耗有实际成本（理智是稀缺资源）
   b) 赤金售卖有机会成本（赤金可以换钱但也有其他用途）
   c) 升级消耗相对"免费"（作战记录容易获得）
   d) 基建合成有材料成本
   按加权代价排序，让"代价最低"的路径排在前面。

3. 路径展示顺序可以更智能
   问题：finalizeResult 中先按种类数排序，再按总数排序。
   但用户可能更关心"哪条路径最容易执行"。

   建议：综合考虑以下因素进行排序：
   a) 步骤总数（越少越好）
   b) 是否需要消耗理智（不需要的排前面）
   c) 是否需要特定干员等级（不需要的排前面）
   d) 涉及的不同操作类型数量（越少越好）


【六、架构层面优化】

1. 前端发送完整 items 数组到后端（严重浪费）
   位置：Transmission.js → server.js → DPnew.js
   问题：前端将 200+ 个物品的完整数据发送到后端，
   每次请求的 payload 约 20~30KB。实际上后端已经有 DataService.js，
   包含完整的物品数据。

   建议：前端只发送 target 值和用户设置（开关状态），
   后端根据开关状态自行过滤物品列表。
   这可以将请求 payload 从 ~30KB 降低到 ~200B。

2. 没有利用 Web Worker 进行并行计算
   问题：当前计算在 Express 的主线程中执行，
   如果计算耗时较长，会阻塞其他请求。

   建议：
   a) 使用 Node.js 的 worker_threads 模块，将计算放到工作线程
   b) 设置计算超时（如 5 秒），超时返回已找到的部分结果
   c) 对于常见差值，预计算结果并缓存

3. 缺少计算超时机制
   问题：如果某个输入导致搜索空间特别大，计算可能耗时很长，
   没有超时保护。

   建议：在主循环中定期检查已用时间，超过阈值后返回当前最优结果。

4. 可以预计算常见差值的结果
   问题：差值范围是 -5000~5000，共 10001 个可能值。
   很多用户查询的差值是相同的（如 ±1、±2、±5、±10 等常见值）。

   建议：
   a) 服务启动时预计算最常见的 100~200 个差值（如 ±1~±100）
   b) 将结果存入持久化缓存（文件或 Redis）
   c) 用户请求时先查缓存，命中则直接返回


【七、代码可维护性】

1. 魔法数字 ID 遍布全文（严重）
   位置：stageIds、tradeGoldIds、materialIds、getMaxCountForId 中的硬编码 ID
   问题：87/88/89/.../119/100~103 等数字没有任何语义，
   阅读代码时完全无法理解每个 ID 代表什么物品。
   修改物品数据时极易遗漏或出错。

   建议：
   a) 在 DataService 中为每个物品定义常量名，如：
      TRADE_GOLD_2 = 117, TRADE_GOLD_3 = 118, TRADE_GOLD_4 = 119
   b) 用常量名替代所有硬编码 ID
   c) 或者改为基于 item.type 的分类判断，而非基于 id

2. savePath 函数参数过多（12 个参数）
   位置：第 166~167 行
   问题：12 个参数难以记忆和维护，调用时容易传错顺序。

   建议：将参数封装为配置对象：
   savePath(dp, sum, path, {
     maxPaths, target, epsilon, items,
     upgrade0Limit, upgrade1Limit, upgrade2Limit,
     sanityLimit, targetPathCount
   })

3. 三个贪心优化器代码高度重复
   位置：getOptimalTradeGoldCombo、getOptimalMaterialCombo、getOptimalStageCombo
   问题：三个函数的逻辑结构几乎相同（贪心从大到小填充），
   只是物品 ID 和面额不同。

   建议：抽取为通用的贪心组合函数：
   getOptimalCombo(subTarget, denominations, cache)
   其中 denominations = [{id, value}, ...] 按 value 降序排列。

4. 第一阶段和第二阶段的物品处理逻辑重复
   位置：第 424~455 行（第一阶段）和第 463~504 行（第二阶段）
   问题：两个阶段中对 tradeGold/material/stage 的特殊处理逻辑
   完全相同，复制粘贴了一遍。

   建议：抽取为 getOptimalFragment(item, count, ...) 函数，
   两个阶段共用。

总结：算法优化优先级排序
================================================================================

第一优先级（修复 bug + 显著提速）：
  □ 构建 Map<id, item> 索引表，替代所有 items.find() 调用
  □ 修复 stageCache 缓存污染 bug（不同用户设置返回错误结果）
  □ 将模块级缓存改为请求级，或加入 items 哈希作为 key
  □ 前端只发送 target + 设置，不发送完整 items 数组

第二优先级（提升搜索效率）：
  □ 恢复或改进 enoughPaths 的 break 逻辑
  □ 动态调整距离剪枝阈值（替代硬编码 3000）
  □ 优化物品扩展范围的启发式条件
  □ 添加计算超时机制（防止极端输入阻塞服务器）

第三优先级（提升结果质量）：
  □ stageCombo 改用精确 DP 替代贪心
  □ 引入路径多样性策略（按类型分组选取）
  □ 引入加权代价函数（区分理智/赤金/升级的实际成本）
  □ 利用物品价值的数学特性进行分层搜索

第四优先级（架构优化）：
  □ 使用 worker_threads 避免阻塞主线程
  □ 预计算常见差值的结果
  □ 实现双向搜索策略

第五优先级（代码可维护性）：
  □ 魔法数字 ID 替换为语义常量
  □ savePath 参数封装为对象
  □ 三个贪心优化器合并为通用函数
  □ 两个阶段的重复逻辑抽取为共用函数
  □ 去掉 epsilon 浮点容差，改用纯整数运算

注意：第一优先级中的"stageCache 缓存污染"是一个实际 bug，
在多用户并发场景下会返回错误结果，应最先修复。
"items.find() 替换为 Map"是投入产出比最高的性能优化，
改动量小但效果显著。
