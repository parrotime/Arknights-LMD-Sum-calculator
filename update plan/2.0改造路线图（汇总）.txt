阶段一：架构重构（2.0 的骨架）
================================================================================

目标：重建项目架构，为后续所有功能改造打好基础。
本阶段改动最大，但不涉及新功能，只是把现有功能迁移到更好的架构上。

1.1 构建工具迁移：CRA → Vite 【前置】
    □ 初始化 Vite + React 项目
    □ 迁移现有源码到 Vite 项目结构
    □ 删除 craco.config.js、react-scripts 相关配置
    □ 确认 dev / build / preview 均正常工作
    【联动：1.5 CSS Modules（Vite 原生支持，迁移时一起做）】
    【来源：角度一-1.1】

1.2 引入 TypeScript（可选，但推荐）
    □ 配置 tsconfig.json
    □ 将核心文件逐步从 .js 改为 .tsx/.ts（可渐进式迁移）
    □ 优先迁移：DataService、DPnew、Transmission
    【联动：1.1 Vite 迁移时一起配置 TS 支持】
    【来源：角度一-1.1】

1.3 数据层统一 【前置】
    □ 将游戏数据从 DataService.js 抽取为独立 JSON 文件
    □ 前后端共用同一份 JSON 数据源（或后端为唯一数据源）
    □ 删除前端 DataService.js 副本
    □ Data.js 页面的表格数据改为从数据源动态生成（不再硬编码）
    【联动：1.4 API 瘦身（数据统一后前端不再发送 items）】
    【联动：阶段四中的"游戏数据更新流程"】
    【来源：角度一-4, 角度二-7.2, 角度四-6】

1.4 API 通信瘦身 【前置】
    □ 前端只发送 { target, settings }（不再发送完整 items 数组）
    □ 后端根据 settings（开关状态）自行过滤物品列表
    □ 简化缓存 key（从 ~30KB JSON 降为 target + settings 哈希）
    □ items 数组的 schema 校验改为后端内部逻辑
    【依赖：1.3 数据层统一完成后才能做】
    【联动：阶段二算法优化中的缓存改造】
    【来源：角度三-6.1, 角度四-1.4, 角度四-1.5】

1.5 CSS 架构迁移
    □ 全局 CSS 改为 CSS Modules（.module.css）
    □ 消除通用类名冲突风险（.title, .input-group 等）
    □ 按组件拆分样式文件
    【联动：1.1 Vite 原生支持 CSS Modules】
    【联动：1.6 组件拆分时同步拆分样式】
    【来源：角度一-8.1, 角度五-5.3】

1.6 组件架构重构
    □ 抽取共享 Layout 组件（Sidebar + Footer + 主内容区）
      → 消除 4 个页面各自定义 Sidebar 的重复
      → Sidebar 使用 NavLink 实现当前页面高亮
    □ 拆分 App.js（981行）为独立组件：

    □ createHeartEffect 改为 React 方式实现（不直接操作 DOM）

================================================================================
阶段二：算法优化（2.0 的核心引擎）
================================================================================

目标：修复算法 bug，提升计算速度，改善结果质量。
本阶段可与阶段一并行推进（算法在后端，架构重构主要在前端）。

2.1 修复算法 bug 【前置】
    □ 修复 stageCache 缓存污染：不同用户的 stageItems 不同但共享缓存
      → 方案：将缓存改为函数内局部变量，或将 stageItems id 列表编入 key
    □ 将模块级缓存（tradeGoldCache/materialCache/stageCache）
      改为每次 findPaths 调用时创建的局部变量，防止内存泄漏
    【来源：角度三-3.2, 角度三-2.2, 角度一-6.3】

2.2 核心性能优化【跳过】
    □ 构建 Map<id, item> 索引表，替代所有 items.find() 的 O(n) 查找
    □ 恢复或改进 enoughPaths 的 break 逻辑（第505~509行被注释的break）
      → 至少恢复 items 循环层的 break，或引入"宽限计数器"
    □ 距离剪枝阈值从硬编码 3000 改为动态值
      → Math.max(Math.abs(target) * 1.5, 1000)
    □ 优化物品扩展范围的启发式条件（第431行和第469行）
    【来源：角度三-2.1, 角度三-3.3, 角度三-4.1, 角度三-4.2】

2.3 结果质量提升【跳过】
    □ stageCombo（消耗理智组合）从贪心改为精确 DP
      → 物品种类不多（十几种），精确求解完全可行
    □ 引入路径多样性策略：按类型分组（升级型/刷图型/合成型/混合型）
      → 保证返回的 10 条路径覆盖不同策略
    □ 引入加权代价函数（理智 > 赤金 > 升级 > 合成）
      → 让"代价最低"的路径排在前面
    【联动：阶段三中的路径摘要信息展示】
    【来源：角度三-5, 角度三-3.1】

2.4 代码可维护性
    □ 魔法数字 ID 替换为语义常量或基于 item.type 的判断
    □ savePath 的 12 个参数封装为配置对象
    □ 三个贪心优化器合并为通用函数 getOptimalCombo()
    □ 两个阶段的重复物品处理逻辑抽取为共用函数
    □ 去掉 epsilon 浮点容差，改用纯整数运算
    【来源：角度三-7】

2.5 后端架构加固
    □ 使用 worker_threads 将计算放到工作线程（避免阻塞主线程）
    □ 添加后端 start 脚本，生产环境使用 PM2 管理进程
    □ 日志规范化：console.log → pino（区分 info/warn/error）
    【联动：1.4 API 瘦身完成后缓存 key 更简洁】
    【来源：角度三-6.2, 角度一-6.4, 角度四-3.4】
