================================================================================
                龙门币凑数计算器 2.0 改造计划 —— 角度一：代码层面分析
================================================================================

【一、构建工具与工程化】

1. 构建配置混乱（严重）
   现状：项目同时存在三套构建配置——
   - CRA (react-scripts) 作为基础
   - craco.config.js 覆盖 CRA 配置（加了 worker-loader）
   - webpack.config.js 又是一套独立的 webpack 配置（输出到 dist/）
   实际运行用的是 craco（package.json scripts 里写的是 craco start/build），
   webpack.config.js 基本是废弃的，但留在项目里造成困惑。

   建议：迁移到 Vite。Vite 开发体验远优于 CRA，配置简洁，
   且 CRA 已经不再积极维护。迁移后删除 craco.config.js 和 webpack.config.js。

2. 前后端依赖混在一起（中等）
   现状：前端 package.json 里包含了后端依赖（cors、express-rate-limit、node-cache），
   虽然后端有自己的 package.json，但前端的 package.json 也装了这些。

   建议：清理前端 package.json 中的后端依赖，保持前后端依赖完全独立。

3. 没有环境变量管理（中等）
   现状：API 地址和 CORS origin 都是硬编码的，切换本地/线上需要手动注释代码。
   代码中有多处"请在此处调试1""请在此处调试2"的注释标记。
   - Transmission.js:16 — fetch URL 手动切换
   - server.js:46 — CORS origin 手动切换

   建议：使用 .env 文件 + 环境变量（VITE_API_URL 等），
   通过 npm scripts 区分 dev/prod 环境，彻底消除手动注释切换。

4. 没有 TypeScript（低优先级，但推荐）
   现状：项目是纯 JS，虽然有一个 DataService.ts 但已标注"弃用"。
   devDependencies 里装了 @types/react 但并未实际使用。

   建议：2.0 改造时全面迁移到 TypeScript。
   特别是 DataService 的数据结构、API 请求/响应类型、reducer action 类型，
   这些地方加上类型后能大幅减少 bug。


【二、冗余文件与死代码】

1. 废弃文件未清理
   - src/BFS.js — 文件头注释"已经弃用 2025.3.1"，但仍留在项目中
   - src/DataService.ts — 注释"已经弃用，后续可能参考"，与 DataService.js 重复
   - webpack.config.js — 实际未被使用（craco 才是真正的构建入口）
   - dist/ 目录 — webpack.config.js 的输出目录，应该也是废弃的

   建议：全部删除。

2. 大量注释掉的 console.log
   现状：几乎每个文件都有被注释掉的 console.log/console.error，
   例如 App.js 中约有 15+ 处，server.js 中约有 10+ 处，DPnew.js 中约有 8+ 处。
   还有一些保留的 console.log 在生产环境中也不应该存在。

   建议：全部清理。如果需要调试日志，使用一个可配置的 logger 工具，
   通过环境变量控制是否输出。

3. 注释掉的旧代码块
   - App.js:229-241 — 旧版 handleToggleChange 被注释保留
   - DPnew.js:505-509 — 注释掉的 break 语句
   - server.js:14-15 — 注释掉的本地调试 URL

   建议：全部删除，版本历史交给 git 管理。


【三、组件架构与状态管理】

1. App.js 巨型组件（严重）
   现状：App.js 共 981 行，MainCalculator 一个组件承担了几乎所有职责——
   输入处理、设置面板、计算逻辑、缓存管理、彩蛋逻辑、路径展示、弹窗、侧边栏。
   这导致代码难以维护、难以测试、难以复用。

   建议：拆分为多个独立组件，例如：
   - InputPanel — 龙门币输入 + 限制条件输入
   - SettingsPanel — 14 个开关设置
   - ResultSection — 计算结果展示
   - Sidebar — 导航栏（目前在 MainCalculator 内部定义，每次渲染都重新创建）
   - EasterEgg — 彩蛋相关逻辑
   - Modal — 通用弹窗组件

2. Sidebar 在组件内部定义（bug 级别）
   现状：App.js:591 Sidebar 作为函数组件定义在 MainCalculator 内部。
   这意味着每次 MainCalculator 重新渲染时，Sidebar 都会被重新创建，
   React 会认为它是一个全新的组件，导致不必要的卸载和重新挂载。

   建议：将 Sidebar 提取为独立的顶层组件。

3. getInitialState 每次都覆盖 localStorage 中的设置（bug）
   现状：App.js:58-71 getInitialState() 虽然从 localStorage 读取了保存的状态，
   但紧接着就用硬编码的值覆盖了所有 settings 字段。
   这意味着用户保存的设置偏好永远不会被恢复——每次刷新页面都会重置为默认值。

   建议：只在 localStorage 中没有保存数据时才使用默认值。

4. localStorage 存储了不该持久化的状态（中等）
   现状：App.js:206-208 的 useEffect 把整个 state 存入 localStorage，
   包括 isCalculating、differenceError、clickCount 等临时状态。
   这些状态不应该被持久化，刷新后应该重置。

   建议：只持久化需要保留的字段（num1、num2、settings、upgradeCount 等）。


【四、已发现的 Bug 与隐患】

1. server.js 重复注册 rate limiter（bug）
   现状：server.js:31 和 server.js:56 对 "/find-paths" 路径注册了两次 apiLimiter。
   这意味着每个请求会被限流中间件检查两次，实际限流效果是 15/2 ≈ 7-8 次/分钟，
   比预期的 15 次/分钟更严格。

   建议：删除其中一处重复注册。

2. HTML 嵌套 <p> 标签（bug）
   现状：App.js:923-935 弹窗中 <p> 标签内部又嵌套了 <p> 标签，
   这是无效的 HTML，浏览器会自动修正但可能导致意外的 DOM 结构。

   建议：将内层 <p> 改为 <span> 或 <div>。

3. footer 中 ICP 备案链接没有 href（低）
   现状：App.js:909 的 <a> 标签没有 href 属性，点击不会跳转到备案查询页面。

   建议：添加正确的备案查询链接。

4. PathRenderer 中的副作用计算（隐患）
   现状：PathRenderer.js:77 在 JSX 的 map 回调中通过 currentLMD += stepValue
   修改外部变量来计算累计龙门币。这是在渲染过程中产生副作用，
   在 React StrictMode 下会因为双重渲染导致数值翻倍。

   建议：在渲染前预先计算好每一步的累计值，而不是在 JSX 中修改变量。

5. handleCalculate 中 paths 可能为 undefined（隐患）
   现状：App.js:460 dispatch SET_PATHS 使用的 paths 变量，
   在 catch 分支中已经 dispatch 了空数组，但 finally 之前的这行代码
   在 paths 为空时仍然会执行，可能导致覆盖 catch 中设置的空数组。

   建议：重构 try/catch 逻辑，确保 paths 的赋值路径清晰。


【五、数据层问题】

1. 前后端 DataService 数据重复（严重）
   现状：src/DataService.js 和 backend/DataService.js 是两份几乎相同的数据文件，
   200+ 条物品数据各维护一份。如果游戏更新需要修改数据，必须同时改两个文件，
   极易遗漏导致前后端数据不一致。

   建议：数据只维护一份。方案有两种：
   a) 后端作为唯一数据源，前端启动时从后端拉取（推荐）
   b) 抽取为共享的 JSON 文件，前后端都从同一文件读取

2. 前端把完整 items 数组发送给后端（性能浪费）
   现状：Transmission.js 每次请求都把过滤后的 items 数组（可能 100-200 条）
   完整序列化发送给后端。每条 item 包含 id、item_id、item_name、item_value、
   rarity、type、consume 共 7 个字段。

   建议：前端只发送 settings 配置（14 个布尔值），
   后端根据 settings 自行过滤，大幅减少请求体积。

3. 缓存 key 包含完整 items 数组（性能问题）
   现状：server.js:147 用 JSON.stringify(items) 作为缓存 key 的一部分，
   导致 key 长度可能达到数千字符，影响缓存查找效率。

   建议：如果改为发送 settings，缓存 key 自然就短了。

4. 物品 ID 使用魔法数字（可维护性差）
   现状：DPnew.js 中大量硬编码的 ID 数组和条件判断——
   - stageIds = [87, 88, 89, ..., 211]
   - tradeGoldIds = [117, 118, 119]
   - materialIds = [100, 101, 102, 103]
   - getMaxCountForId 中 [1,2,3,4,5,6,7,8,21,22,...] 等
   - id >= 150 && id <= 180、id >= 181 && id <= 209 等范围判断
   这些数字没有任何语义，维护时完全无法理解含义。

   建议：在数据源中为每个 item 增加 maxCount 字段，
   或者按 type 分组定义 maxCount 规则，消除魔法数字。


【六、后端代码质量】

1. DPnew.js 中大量 items.find() 查找（性能问题）
   现状：savePath、normalizePath、finalizeResult 等函数中频繁使用
   items.find(i => i.id === step.id) 进行线性查找。
   在路径计算的热循环中，这是 O(n) 的操作，items 有 200+ 条，
   每次计算可能执行数千次 find。

   建议：在 findPaths 入口处构建一个 Map<id, item>，
   后续所有查找改为 O(1) 的 Map.get()。

2. savePath 函数参数过多（可读性差）
   现状：DPnew.js:166 savePath 函数有 12 个参数：
   (dp, sum, path, maxPaths, target, epsilon, items,
    upgrade0Limit, upgrade1Limit, upgrade2Limit, sanityLimit, targetPathCount)
   调用处也非常冗长，难以阅读。

   建议：将相关参数打包为对象，例如：
   savePath(dp, sum, path, { maxPaths, target, epsilon, items, limits, targetPathCount })

3. 模块级缓存无限增长（内存泄漏风险）
   现状：DPnew.js:3-5 定义了三个模块级 Map 缓存：
   tradeGoldCache、materialCache、stageCache。
   这些缓存在 Node.js 进程生命周期内永远不会被清理，
   随着请求增多会持续增长。

   建议：每次 findPaths 调用前清空缓存，或改为函数内局部变量。

4. 后端缺少 start 脚本和进程管理
   现状：backend/package.json 没有 start 脚本，
   也没有使用 pm2 或 nodemon 等进程管理工具。

   建议：添加 start 脚本，生产环境使用 pm2 管理进程。


【七、前端代码质量】

1. 直接操作 DOM（反模式）
   现状：App.js:563-583 createHeartEffect 函数直接使用
   document.createElement、document.body.appendChild 来创建爱心动画元素。
   这绕过了 React 的虚拟 DOM，是 React 中的反模式。

   建议：用 React state 管理动画元素列表，通过 JSX 渲染，
   或使用 CSS animation + React 状态控制。

2. handleCalculate 职责过重（可维护性差）
   现状：App.js:287-524 handleCalculate 函数长达 237 行，承担了：
   - 彩蛋检测与触发
   - 输入验证
   - 物品过滤
   - 缓存读取
   - API 调用
   - 错误处理
   - 历史记录更新
   - 缓存写入

   建议：拆分为多个独立函数，每个函数只做一件事。
   例如 validateInput()、filterItems()、checkCache()、callAPI() 等。

3. useCallback 依赖项问题（隐患）
   现状：handleCalculate 的依赖项包含 state.history，
   这意味着每次历史记录变化都会重新创建这个函数。
   而 history 在每次计算后都会变化，所以 useCallback 的缓存几乎无效。

   建议：使用 useRef 存储 history，或者将历史记录更新逻辑移出 handleCalculate。

4. 设置项的命名不一致（可读性差）
   现状：14 个设置项中，有的用 disable 前缀（disable3Star），
   有的用 enable 前缀（enableUpgradeOnly0）。
   disable 表示"关闭某功能"，enable 表示"开启某功能"，
   混用导致理解开关状态时需要反复思考。

   建议：统一命名方向，全部改为正向语义（如 allow3Star: true）。


【八、CSS 与样式】

1. 全局 CSS 无隔离（潜在冲突）
   现状：所有 CSS 文件都是全局作用域，类名如 .title、.input-group 等
   非常通用，随着项目增长容易产生样式冲突。

   建议：迁移到 CSS Modules（.module.css）或 Tailwind CSS，
   实现样式隔离。如果迁移 Vite，CSS Modules 开箱即用。

2. webpack.config.js 中的中文注释乱码
   现状：webpack.config.js 中的中文注释全部显示为乱码（如"ƥ������"），
   说明文件编码有问题。虽然这个文件建议删除，但反映出项目缺少
   统一的编码规范（.editorconfig）。

   建议：添加 .editorconfig 文件统一编码为 UTF-8。


================================================================================
总结：代码层面改造优先级排序
================================================================================

第一优先级（基础设施，后续改造的前提）：
  □ 迁移 CRA → Vite + TypeScript
  □ 清理废弃文件（BFS.js、DataService.ts、webpack.config.js、dist/）
  □ 引入环境变量管理，消除手动注释切换
  □ 清理所有注释掉的 console.log 和死代码

第二优先级（修复 bug 和明确的错误）：
  □ 修复 server.js 重复注册 rate limiter
  □ 修复 getInitialState 覆盖 localStorage 设置的问题
  □ 修复 HTML 嵌套 <p> 标签
  □ 修复 PathRenderer 中渲染时的副作用计算
  □ 修复 handleCalculate 中 paths 变量的控制流问题

第三优先级（架构重构，提升可维护性）：
  □ 拆分 App.js 巨型组件为多个独立组件
  □ 统一前后端数据源，消除 DataService 重复
  □ 前端只发送 settings 而非完整 items 数组
  □ 消除 DPnew.js 中的魔法数字
  □ 统一设置项命名方向（disable/enable 混用问题）

第四优先级（代码质量提升）：
  □ DPnew.js 中 items.find() 改为 Map 查找
  □ savePath 参数重构为对象
  □ 修复模块级缓存无限增长问题
  □ handleCalculate 拆分为多个小函数
  □ createHeartEffect 改为 React 方式实现
  □ localStorage 只持久化必要字段
  □ CSS 迁移到 CSS Modules
  □ 清理前端 package.json 中的后端依赖
  □ 添加 .editorconfig

注意：以上优先级是从"代码层面"角度排列的。
实际执行时需要结合角度二（用户体验）和角度三（算法优化）综合考虑。
例如如果 2.0 要做 UI 大改，那么组件拆分和 CSS 迁移可以合并到 UI 重构中一起做。
